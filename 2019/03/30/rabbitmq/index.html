    消息队列RabbitMQ

    RabbitMQ基于 ErLang 语言,开发具有高可用高并发的优点,适合集群服务器.它有消息确认机制和持久化机制,可靠性高.

    简介

    消息队列一般解决异步通信和服务间解耦的问题(通知短信,统计数据等),RabbitMQ为AMQP,使用ErLang语言编写.ErLang语言可以做到公平调度.

    RabbitMQ的基本结构

    消息生产者Producer

    Put消息到队列中.

    消息消费者Comsumer

    采用订阅的方式获取消息.

    Queue

    消息队列提供FIFO(先进先出)的处理机制,队列可以设置为持久化,临时,和自动删除

    • 持久化

      消息会在服务器本地硬盘存储,防止数据丢失

    • 临时

      消息存储于内存,系统重启后会消失

    • 自动删除

      当没有用户连接到服务器时,队列中的消息会被自动删除

    ExChange

    ExChange提供消息路由策略.在RabbitMQ中,生产者(Producer)会先把消息发送给ExChange.一个ExChange可以与多个Queue进行绑定,Exchange根据生产者(Produce)传来消息中的routing_key将消息传给指定的Queue,对于routing_key有特定的路由算法.Exchange也可以设置为持久化,临时和自动删除.

    ExChange的类型

    • direct(直接交换器)

      这是Exchange默认的交换模式,消息会被路由到与routing_key相同的Queue,即key就是Queue.

    • fanout(广播式交换器)

      不检查routing_key,对绑定的Queue全部都发送一遍消息.

    • topic(主题交换器)

      类似于组播方式,会将消息转发给与routing_key匹配的所有队列(模糊匹配),类似于搜索中的模糊匹配原则.例如,key为user.msg的消息会被user.msg,*.msg,*.*,#.user.msg.#的队列所匹配.

    • headers(头部匹配交换器)

      通过查看消息的头部数据进行匹配(类似于http的header信息)

    Binding

    将一个特定的 ExChange 和一个特定的 Queue 绑定起来.ExChange 和 Queue 的绑定可以是多对多的关系.

    Virtual Host(虚拟机)

    类似于Tomcat或Nginx,在 RabbitMQ Server 上可以创建多个虚拟的 Message Broker,又叫做 Virtual Hosts(vhosts).每一个 vhost 本质上也是一个server,分别管理各自的 ExChange,和 bindings.vhost 相当于物理的 Server,可以为不同 app 提供边界隔离,使得应用安全的运行在不同的 vhost 实例上,相互之间不会干扰.Producer 和 Consumer 连接 rabbit server 需要指定一个 vhost.

    基本使用流程

    • 客户端连接到消息队列服务器,打开一个 Channel.
    • 客户端声明一个 ExChange,并设置相关属性.
    • 客户端声明一个 Queue,并设置相关属性.
    • 客户端使用 Routing Key,在 ExChange 和 Queue 之间建立好绑定关系.
    • 客户端投递消息到 ExChange.
    • ExChange 接收到消息后,就根据消息的 key 和已经设置的 binding,进行消息路由,将消息投递到一个或多个队列里.
